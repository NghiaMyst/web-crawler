# Plan 05: Wire Guards into crawlWorker + Retry and Dead-Letter Verification

## Goal
Wire all four service modules (Bloom Filter, politeness, robots.txt, content hash) into the crawl pipeline, and verify the existing BullMQ retry/dead-letter configuration satisfies CRAWL-08 and CRAWL-09.

## Context
This is a Wave 2 plan that depends on Plans 01-04 (all four service modules must exist). It modifies two existing files: `crawlWorker.ts` (add pre/post-fetch guards) and `crawlProducer.ts` (add Bloom Filter check at enqueue time). Per D-05, BullMQ retry is already configured correctly -- this plan just verifies it. Per D-06, the BullMQ failed set with `removeOnFail: { count: 500 }` serves as the dead-letter queue.

2. Inside the worker job handler, BEFORE the strategy dispatch (`if (strategy === 'cheerio')`), add guards in this order per RESEARCH.md Pattern 5:

   **Step 1 - Extract domain:**
   ```typescript
   const { hostname } = new URL(url);
   ```

   **Step 2 - Politeness guard (D-01):**
   ```typescript
   await enforcePoliteness(hostname);
   ```

   **Step 3 - robots.txt check (D-03):**
   ```typescript
   const allowed = await isUrlAllowed(url);
   if (!allowed) {
     logger.warn('URL disallowed by robots.txt -- skipping', { url, sourceId, jobId: job.id });
     return; // Skip fetch entirely, job completes as "done" (not a failure)
   }
   ```
   NOTE per D-03: Disallowed URLs are logged with `disallowed` status. The job COMPLETES (returns), it does not throw. This is intentional -- a disallowed URL is not a failure, it's a policy skip.

   NOTE per RESEARCH.md: Do NOT mark disallowed URLs in the Bloom Filter. The Bloom Filter is checked at the producer level (Task 1 above), not here.

3. AFTER the strategy dispatch (after cheerio/playwright fetch completes), add the content hash check (D-04). This requires capturing the response body. Modify the cheerio and playwright branches to capture the result:

   ```typescript
   let responseBody: string | undefined;

   if (strategy === 'cheerio') {
     const result = await cheerioFetch(url, sourceId, job.id ?? 'unknown');
     responseBody = result.rawHtml;
   } else if (strategy === 'playwright') {
     const result = await playwrightFetch(url, sourceId, job.id ?? 'unknown');
     responseBody = result.html;
   }

   // Content hash dedup (D-04) -- skip if unchanged
   if (responseBody && sourceId) {
     const changed = await isContentChanged(sourceId, responseBody);
     if (!changed) {
       logger.info('Content unchanged -- skipping', { url, sourceId, jobId: job.id });
       return;
     }
   }
   ```

4. Per D-05: Do NOT modify `crawlQueue.ts` -- retry config (attempts: 3, exponential 5s) is already correct.
5. Per D-06: Do NOT modify `removeOnFail: { count: 500 }` -- this IS the dead-letter retention.

6. Run type check: `pnpm --filter @web-crawler/crawler run type-check`

## Threat Model

| Boundary | Description |
|----------|-------------|
| crawlWorker guard chain | All HTTP fetches must pass through politeness + robots.txt checks |

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-02-06 | Elevation of Privilege (rate limit bypass) | crawlWorker.ts | mitigate | Guard chain is mandatory in the single crawlWorker code path; source-specific workers (Plan 06) each call enforcePoliteness independently |
| T-02-07 | Spoofing (SSRF) | crawlWorker.ts | accept | URLs come from internal job queue, not direct user input; personal project with configured sources only |

### Must-Haves
- Guard execution order: politeness -> robots.txt -> fetch -> content hash
- Disallowed URLs complete without error (not thrown)
- Unchanged content is skipped after fetch
- crawlQueue.ts retry/DLQ config is NOT modified (D-05, D-06)
- Bloom Filter dedup happens at producer level, not worker level

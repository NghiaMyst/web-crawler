1. Create `apps/api.Tests/Endpoints/HealthEndpointTests.cs`:
   Since the health check is an inline lambda in Program.cs (not a static method in an endpoint class), test it by extracting the health logic into a testable static method.

   Create a helper class `HealthCheck` in the `WebCrawlerApi.Endpoints` namespace:
   ```csharp
   // In a new file apps/api/Endpoints/HealthCheck.cs
   using Microsoft.EntityFrameworkCore;
   using StackExchange.Redis;
   using WebCrawlerApi.Data;

   namespace WebCrawlerApi.Endpoints;

   public static class HealthCheck
   {
       internal static async Task<IResult> CheckHealth(AppDbContext db, IConnectionMultiplexer redis)
       {
           string pgStatus, redisStatus;

           try
           {
               await db.Database.ExecuteSqlRawAsync("SELECT 1");
               pgStatus = "ok";
           }
           catch
           {
               pgStatus = "error";
           }

           try
           {
               await redis.GetDatabase().PingAsync();
               redisStatus = "ok";
           }
           catch
           {
               redisStatus = "error";
           }

           var overall = (pgStatus == "ok" && redisStatus == "ok") ? "ok" : "degraded";
           var body = new { status = overall, postgres = pgStatus, redis = redisStatus };

           return overall == "ok" ? Results.Ok(body) : Results.Json(body, statusCode: 503);
       }
   }
   ```

   Then update Program.cs to replace the existing health endpoint:
   ```csharp
   // Replace:
   app.MapGet("/health", () => Results.Ok(new { status = "ok", service = "api" }));
   // With:
   app.MapGet("/health", HealthCheck.CheckHealth);
   ```

   Add `using WebCrawlerApi.Endpoints;` if not already present (should be from Plan 01).

   For tests:
   - Use EF Core InMemory provider for AppDbContext. Note: `ExecuteSqlRawAsync("SELECT 1")` will throw `InvalidOperationException` on InMemory provider since it doesn't support raw SQL. This is actually useful for testing the "postgres error" case.
   - For the "postgres ok" case, mock the DbContext's `Database` facade or use a real PostgreSQL test connection. ALTERNATIVE: extract the health logic to accept two `Func<Task>` delegates for testability.

   RECOMMENDED APPROACH for testability: Make the health check logic accept `Func<Task>` probes:
   ```csharp
   internal static async Task<IResult> CheckHealth(
       Func<Task> pgProbe, Func<Task> redisProbe)
   {
       string pgStatus, redisStatus;
       try { await pgProbe(); pgStatus = "ok"; } catch { pgStatus = "error"; }
       try { await redisProbe(); redisStatus = "ok"; } catch { redisStatus = "error"; }
       var overall = (pgStatus == "ok" && redisStatus == "ok") ? "ok" : "degraded";
       var body = new { status = overall, postgres = pgStatus, redis = redisStatus };
       return overall == "ok" ? Results.Ok(body) : Results.Json(body, statusCode: 503);
   }
   ```

   Then in Program.cs:
   ```csharp
   app.MapGet("/health", async (AppDbContext db, IConnectionMultiplexer redis) =>
       await HealthCheck.CheckHealth(
           () => db.Database.ExecuteSqlRawAsync("SELECT 1"),
           () => redis.GetDatabase().PingAsync().AsTask()));
   ```

   Tests can then call `HealthCheck.CheckHealth(pgProbe, redisProbe)` with:
   - Both succeed: `() => Task.CompletedTask`
   - PG fails: `() => throw new Exception("pg down")`
   - Redis fails: `() => throw new Exception("redis down")`

   Add the file `apps/api/Endpoints/HealthCheck.cs` to `files_modified` in Plan 01's frontmatter is not needed since this plan owns the creation.

# Phase 11 Plan 02 Summary: FTS Migration

## One-liner
PostgreSQL FTS infrastructure: SearchConfig entity, NpgsqlTsVector column on DataEntry, GIN index, PL/pgSQL trigger, and 5 seed rows via raw SQL migration.

## What Was Done

- Created `SearchConfig` entity (`apps/api/Data/Entities/SearchConfig.cs`) with `SourceId` (UUID PK, FK to sources.id) and `JsonPaths` (string[], maps to TEXT[])
- Added `NpgsqlTsVector? SearchVector` property to `DataEntry` entity with `using NpgsqlTypes;`
- Updated `AppDbContext` with `DbSet<SearchConfig>`, `HasColumnType("tsvector")` for SearchVector, and full SearchConfig entity mapping (`text[]`, one-to-one FK with CASCADE)
- Generated migration scaffold via `dotnet ef migrations add AddFtsSearchVector` then replaced auto-generated CreateTable/AddColumn with raw SQL `migrationBuilder.Sql()` calls

## Migration Contents

- `ALTER TABLE data_entries ADD COLUMN search_vector tsvector`
- `CREATE INDEX ix_data_entries_search_vector ON data_entries USING GIN (search_vector)`
- `CREATE TABLE search_configs (source_id UUID PRIMARY KEY REFERENCES sources(id) ON DELETE CASCADE, json_paths TEXT[] NOT NULL DEFAULT '{}')`
- 5 INSERT seed rows (football, genshin, lol, anilist, mangadex) using `SELECT id FROM sources WHERE parser_key = '...'` with `ON CONFLICT DO NOTHING`
- PL/pgSQL function `data_entries_search_vector_update()` reading `search_configs.json_paths` for `NEW.source_id`, extracting via `jsonb_path_query`, concatenating, writing `to_tsvector('english', ...)`, falling back to `entry_key` when empty
- `BEFORE INSERT OR UPDATE` trigger `data_entries_search_vector_trigger` on `data_entries`

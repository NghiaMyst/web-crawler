    Create the `.github/` directory (if it does not exist) and `.github/workflows/` subdirectory, then create `.github/workflows/deploy.yml` with this exact content:

    ```yaml
    # .github/workflows/deploy.yml
    # Builds crawler + api Docker images in parallel, pushes to GCP Artifact Registry.
    # Deploy job (SSH to GCE) is in the same file and runs after both images push.
    # Triggered: push to main + manual workflow_dispatch.
    #
    # Required GitHub Secrets (see .github/secrets.md):
    #   GCP_PROJECT_ID, GCP_WORKLOAD_IDENTITY_PROVIDER, GCP_SERVICE_ACCOUNT
    #   GCE_SSH_PRIVATE_KEY, GCE_SSH_HOST, GCE_SSH_USER, GCE_SSH_KNOWN_HOSTS
    name: Build, Push, Deploy

    on:
      push:
        branches: [main]
      workflow_dispatch:

    jobs:
      build-and-push:
        runs-on: ubuntu-latest
        permissions:
          contents: read
          id-token: write  # Required for Workload Identity Federation

        strategy:
          matrix:
            include:
              - app: crawler
                dockerfile: apps/crawler/Dockerfile
              - app: api
                dockerfile: apps/api/Dockerfile

        steps:
          - uses: actions/checkout@v4

          - name: Authenticate to Google Cloud
            id: auth
            uses: google-github-actions/auth@v3
            with:
              token_format: access_token
              project_id: ${{ secrets.GCP_PROJECT_ID }}
              workload_identity_provider: ${{ secrets.GCP_WORKLOAD_IDENTITY_PROVIDER }}
              service_account: ${{ secrets.GCP_SERVICE_ACCOUNT }}

          - name: Set up Docker Buildx
            uses: docker/setup-buildx-action@v4.1.0

          - name: Log in to Artifact Registry
            uses: docker/login-action@v4.2.0
            with:
              registry: asia-southeast1-docker.pkg.dev
              username: oauth2accesstoken
              password: ${{ steps.auth.outputs.access_token }}

          - name: Build and push ${{ matrix.app }}
            uses: docker/build-push-action@v7.2.0
            with:
              context: .
              file: ${{ matrix.dockerfile }}
              platforms: linux/amd64
              push: true
              tags: |
                asia-southeast1-docker.pkg.dev/${{ secrets.GCP_PROJECT_ID }}/webcrawler/${{ matrix.app }}:${{ github.sha }}
                asia-southeast1-docker.pkg.dev/${{ secrets.GCP_PROJECT_ID }}/webcrawler/${{ matrix.app }}:latest
              cache-from: type=registry,ref=asia-southeast1-docker.pkg.dev/${{ secrets.GCP_PROJECT_ID }}/webcrawler/${{ matrix.app }}:cache
              cache-to: type=registry,ref=asia-southeast1-docker.pkg.dev/${{ secrets.GCP_PROJECT_ID }}/webcrawler/${{ matrix.app }}:cache,mode=max

      # deploy job is added in Plan 03 (depends on build-and-push completing)
    ```

    **Key design decisions encoded in this file:**
    - `platforms: linux/amd64` only — GCE VM is AMD64 (e2-medium), halves build time vs multi-arch
    - `cache-from`/`cache-to` use `type=registry` — avoids 10GB GitHub Actions cache limit that the 1.5GB Playwright image would exhaust
    - `mode=max` caches ALL layers including the builder stage — critical for the `mcr.microsoft.com/playwright:v1.50.1-noble` base
    - `oauth2accesstoken` login pattern — required for Artifact Registry when using WIF (not a password, it's a short-lived OAuth2 token)
    - `id-token: write` permission — required on the job-level for WIF to exchange GitHub OIDC token for GCP access token
    - `workflow_dispatch` trigger — allows manual re-runs without a code push (useful for first-time testing)

    Do NOT add a `deploy` job yet — that is Plan 03's task. The `# deploy job is added in Plan 03` comment is intentional.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GCP Artifact Registry | Short-lived OAuth2 access token via WIF; expires after job completion |
| GitHub Actions → GCE VM | SSH with Ed25519 key (handled in Plan 03) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-02-01 | Elevation of Privilege | WIF token exchange | mitigate | `--attribute-condition="assertion.repository=='NghiaMyst/web-crawler'"` on provider (set in Plan 01 gcloud commands) ensures only this repo can exchange tokens. Job-level `id-token: write` permission follows principle of least privilege. |
| T-12-02-02 | Tampering | Docker image push | accept | Images are tagged with `github.sha` providing an immutable audit trail. Artifact Registry does not allow overwriting existing SHA-tagged images by default. |
| T-12-02-03 | Information Disclosure | Build logs | accept | `github.sha`, `GCP_PROJECT_ID` (non-secret) appear in logs. Actual secrets (`GCP_WORKLOAD_IDENTITY_PROVIDER`, `GCP_SERVICE_ACCOUNT`) are masked by GitHub Actions automatically. |
| T-12-02-04 | Denial of Service | Registry cache poisoning | accept | `type=registry` cache tag (`:cache`) is pushed by CI only. If corrupted, rebuilding from scratch is the fallback — `cache-from` failure is non-fatal (BuildKit continues without cache). |
</threat_model>
